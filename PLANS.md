# PLANS.md — 実装計画の暴走防止ガイド

## 最重要指示

実装プランを作成するとき、次のルールを必ず守ってください。

1. **最小で完了可能な計画を作る。** 目的達成に不要な作業は入れない。
2. **計画と実装を分離する。** ユーザーが明示するまでコード変更やファイル変更を開始しない。
3. **タスク分解は有限にする。** 原則として `Phase -> Task` の2階層まで。Taskの下にSubtaskを作らない。
4. **各タスクは検証可能にする。** すべてのタスクに受入条件と検証方法を書く。
5. **後続タスクを乱発しない。** 発見した追加作業は、必須か任意かを分類し、上限内で扱う。
6. **将来の仮定で設計しない。** いま要求されていない柔軟性、抽象化、設定、拡張ポイントを入れない。
7. **曖昧さは制限付きで扱う。** 不明点は最大3件まで。可能なら明示的な仮定を置いて先に進める。
8. **最後に次の一手を1つだけ提示する。** 複数の「次にやること」リストを作らない。

---

## ハードリミット

ユーザーが明示的に上書きしない限り、以下を上限とします。

| 項目 | 上限 |
|---|---:|
| Phase数 | 4 |
| Task数 | 8 |
| ネスト深度 | 2階層まで |
| 1 PhaseあたりのTask数 | 4 |
| Assumption数 | 5 |
| Open Question数 | 3 |
| Blocker / Decision Needed数 | 3 |
| Optional / Parking Lot項目 | 3 |
| 最後に提示するNext Action | 1 |

### 上限を超えそうな場合

上限を超えそうな場合、AIは次の順で対処します。

1. 目的達成に不要な作業を削る。
2. 小さすぎる作業を統合する。
3. MVPに必要な範囲だけを残す。
4. 残りは `Out of Scope` に移す。
5. それでも足りない場合だけ、ユーザーにスコープ選択を求める。

**禁止:** 上限を超えた分を `Future Work`、`Follow-up Tasks`、`Later` などとして大量に列挙してはいけません。

## タスク粒度の基準

タスクは「小さすぎず、大きすぎず、検証可能」である必要があります。

### 良いTaskの条件

1つのTaskは、原則として次を満たします。

- 1つの目的を持つ
- 1つの成果物または明確な変更結果を持つ
- 1人または1つのAIセッションで実行できる
- 1つのPRまたは1つのまとまった編集単位に収まる
- 受入条件が明確
- 検証方法が明確
- 依存関係が明確

### 分割を止める条件

以下を満たすなら、それ以上分解してはいけません。

- 何を変更するかが明確である
- 完了条件が明確である
- 検証コマンド、テスト観点、確認方法のいずれかがある
- 追加分解しても、単なる作業手順やキーストロークの列挙になる

### 分割してよい条件

次の場合だけ、タスクを分割してよいです。

- 独立して実装・レビューできる
- 失敗時の切り戻し単位が異なる
- 異なる専門領域、ファイル群、責務に分かれる
- 受入条件が明確に分かれる
- 片方を完了しても価値がある

### 分割してはいけない例

次のような分解は禁止します。

```text
Task: ログイン画面を修正する
  Subtask: ファイルを開く
  Subtask: 該当箇所を探す
  Subtask: コードを書く
  Subtask: 保存する
  Subtask: テストする
```

これは単なる作業手順であり、独立したタスクではありません。

## 計画作成アルゴリズム

AIは実装プランを作るとき、必ず次の順で考えます。

1. **Goalを1文で再定義する**
   - 何を達成するのかを簡潔に書く。

2. **Non-goalsを明示する**
   - 今回やらないことを書く。
   - スコープ膨張を防ぐため、Non-goalsは必ず書く。

3. **Definition of Doneを先に決める**
   - どの状態なら完了とみなすかを定義する。
   - 完了条件がないタスクは作らない。

4. **最小経路を選ぶ**
   - 目的達成に最短で必要な変更だけを採用する。
   - 便利そうな改善、一般化、将来対応は採用しない。

5. **Taskに分解する**
   - `Phase -> Task` の2階層まで。
   - Task数は最大8。

6. **各Taskに検証を付ける**
   - テスト、型チェック、lint、手動確認、ログ確認など。

7. **Blockerだけを分離する**
   - 実行不能な不明点だけ `Decision Needed` に入れる。
   - 推測で進められるものはAssumptionにする。

8. **停止する**
   - 上記が揃ったら、追加のアイデアを出さずに計画を終了する。

## 必須出力フォーマット

AIが実装プランを出すときは、原則としてこの形式だけを使います。

```md
# Implementation Plan

## Goal
<1文で目的を書く>

## Non-goals
- <今回やらないこと>

## Assumptions
- <仮定。最大5件>

## Definition of Done
- <完了条件>

## Plan
| ID | Task | Outcome | Files / Areas | Acceptance Criteria | Validation | Depends On |
|---|---|---|---|---|---|---|
| T1 | ... | ... | ... | ... | ... | ... |

## Decision Needed / Blockers
- <最大3件。なければ "None" と書く>

## Out of Scope
- <最大3件。なければ "None" と書く>

## Single Next Action
<次に実行すべき1つの行動だけを書く>
```

### フォーマット上の禁止事項

以下の見出しは、ユーザーが明示的に求めない限り使わないでください。

- `Future Work`
- `Follow-up Tasks`
- `Nice to Have`
- `Potential Improvements`
- `Additional Considerations`
- `Long-term Roadmap`

これらは後続タスクの増殖源になりやすいため、必要な場合は `Out of Scope` に最大3件だけ記載します。

## 受入条件の書き方

すべてのTaskにはAcceptance Criteriaを付けます。

### 良いAcceptance Criteria

- 観測可能である
- Yes / Noで判定できる
- ユーザー価値または機能要件に結びついている
- テストまたは手動確認に落とせる
- 実装方法ではなく、期待される結果を書く

### 悪いAcceptance Criteria

```text
- いい感じにする
- 必要に応じて改善する
- リファクタリングする
- テストを追加する
- 品質を上げる
```

### 良い例

```text
- 未ログインユーザーが /settings にアクセスした場合、/login にリダイレクトされる
- 既存のログイン済みユーザーの /settings 表示は変わらない
- 関連する認可テストが成功する
```

## 後続タスクの作成ポリシー

AIは、原則として後続タスクを作ってはいけません。

ただし、次のいずれかに該当する場合のみ、追加作業として記録できます。

1. 受入条件を満たすために必須である
2. 実装前提が崩れ、ユーザー判断が必要である
3. セキュリティ、データ損失、重大な回帰のリスクがある
4. 外部API、DBスキーマ、認証設定などの未確定事項が実行をブロックしている
5. ユーザーが明示的に追加検討を求めた

### 追加作業の分類

追加作業を見つけた場合は、必ず次のどれかに分類します。

| 分類 | 意味 | 扱い |
|---|---|---|
| Blocker | これがないと計画を実行できない | `Decision Needed / Blockers` に入れる |
| Required | 受入条件達成に必須 | 既存Taskに統合する |
| Optional | 目的達成には不要 | 原則として書かない。必要なら `Out of Scope` に最大3件 |
| Out of Scope | 今回の依頼外 | `Out of Scope` に最大3件 |

### 後続タスクにしてはいけないもの

以下は後続タスクとして作成してはいけません。

- ついでのリファクタリング
- 将来の拡張性のための抽象化
- 今回触らないファイルのドキュメント整備
- 「あとで全体的にきれいにする」
- 「必要なら監視を追加する」
- 「パフォーマンスを改善する」だけの曖昧な作業
- 具体的な不具合や受入条件に紐づかないテスト追加

## 過剰実装の抑制ルール

AIは、実装プラン内で次をしてはいけません。

### Scope

- 要求されていない機能を追加しない
- 要求されていない設定項目を追加しない
- バグ修正に無関係な周辺コードを整理しない
- 単純な変更をフレームワーク化しない

### Documentation

- 変更していないコードにdocstringやコメントを追加しない
- 自明な処理にコメントを足さない
- ドキュメント更新は、ユーザー影響または運用影響がある場合だけ行う

### Defensive Coding

- 起こり得ないケースの防御コードを追加しない
- 内部境界ではなく、ユーザー入力・外部API・DB・ファイルI/Oなどの境界で検証する
- 「念のため」のfallbackを増やさない

### Abstraction

- 1回しか使わないhelper、utility、adapterを作らない
- 将来要件のためにinterfaceを増やさない
- 既存パターンがない場所に新しい設計パターンを導入しない

---

## 調査タスクの扱い

`Investigate`、`Research`、`Explore` は後続タスク増殖の起点になりやすいため、制限します。

### 調査タスクを作ってよい場合

- 実装方針を決めるために事実確認が必須
- 外部仕様、既存コード、DBスキーマ、API契約の確認が必要
- 調査結果が明確な成果物になる

### 調査タスクの必須成果物

調査タスクには必ず次を含めます。

```text
調査対象:
決めたいこと:
成果物:
調査を終える条件:
調査後に実装へ進む条件:
```

### 禁止される調査タスク

```text
- 全体を調査する
- 影響範囲を確認する
- 必要に応じて調べる
- より良い方法を検討する
```

これらは終わりがないため禁止です。

## Definition of Ready

実装に移る前に、計画は次を満たしている必要があります。

- Goalが1文で説明されている
- Non-goalsが書かれている
- Task数が8以下である
- ネストが2階層以下である
- 各TaskにAcceptance Criteriaがある
- 各TaskにValidationがある
- Blockerが最大3件に収まっている
- Single Next Actionが1つだけである
- Future WorkやFollow-up Tasksが大量に列挙されていない

これを満たさない計画は、実装に移してはいけません。

---

## Definition of Done

実装完了時は、次を満たす必要があります。

- すべてのTaskのAcceptance Criteriaを満たしている
- 指定されたValidationを実行し、結果を報告している
- 新しい未承認TODOを作っていない
- 不要な一時ファイル、helper script、調査メモを残していない
- 要求外のリファクタリングや抽象化をしていない
- 追加で見つけた作業を、勝手に実装せず分類して報告している
- 次に実行すべき行動を1つだけ提示している

---

## 悪いプラン例

```md
## Plan
1. 要件を整理する
2. コードベースを調査する
3. アーキテクチャを検討する
4. 必要なファイルを修正する
5. テストを追加する
6. ドキュメントを書く
7. 将来的に監視を追加する
8. 必要に応じてパフォーマンスを改善する
9. 後でリファクタリングする
```

### なぜ悪いか

- 完了条件がない
- 受入条件がない
- 検証方法がない
- 調査や検討が終わらない
- 後続タスクが曖昧
- 今回やらない改善が混ざっている

---

## 良いプラン例

```md
# Implementation Plan

## Goal
設定画面への未ログインアクセスをログイン画面へリダイレクトする。

## Non-goals
- 認証方式の変更はしない
- UIデザインの変更はしない
- 他ページのアクセス制御は変更しない

## Assumptions
- 既存の認証状態判定関数を利用できる

## Definition of Done
- 未ログインユーザーは /settings から /login にリダイレクトされる
- ログイン済みユーザーは従来通り /settings を表示できる
- 関連テストが成功する

## Plan
| ID | Task | Outcome | Files / Areas | Acceptance Criteria | Validation | Depends On |
|---|---|---|---|---|---|---|
| T1 | settings routeに認証ガードを追加する | 未ログイン時に/loginへ遷移する | routes/settings, auth guard | 未ログイン時は/loginへリダイレクトされる | route test | None |
| T2 | 既存挙動の回帰テストを追加する | ログイン済み表示が保たれる | settings route tests | ログイン済みユーザーは/settingsを表示できる | test suite | T1 |

## Decision Needed / Blockers
None

## Out of Scope
None

## Single Next Action
T1から実装する。
```

## 計画レビュー用チェックリスト

AIが出した計画は、次のチェックリストでレビューします。

```md
- [ ] Goalは1文か
- [ ] Non-goalsがあるか
- [ ] Task数は8以下か
- [ ] ネストは2階層以下か
- [ ] 各TaskにAcceptance Criteriaがあるか
- [ ] 各TaskにValidationがあるか
- [ ] 調査タスクに終了条件があるか
- [ ] Future Work / Follow-up Tasks が乱発されていないか
- [ ] Optional項目は最大3件か
- [ ] Single Next Actionは1つだけか
- [ ] 要求外の抽象化、リファクタ、設定追加がないか
```

---

## ユーザーが上限を上書きする場合

より詳細な計画が必要な場合、ユーザーは明示的に上限を指定できます。

例:

```text
PLANS.mdの制約は維持しつつ、今回は最大15タスクまで許可します。
ただしネストは2階層まで、Future Workは禁止のままにしてください。
```

AIは、上書きされた項目以外の制約を維持します。

---

## 更新ルール

このファイルは、AIの計画がまだ膨らみすぎる場合にだけ更新します。

更新するときは、次のどれかを変更してください。

- Task数の上限を下げる
- Optional / Parking Lotの上限を下げる
- 禁止見出しを増やす
- Definition of Doneを厳しくする
- 調査タスクの終了条件を具体化する

**原則:** ルールを増やす前に、まず上限と出力形式を厳しくします。
